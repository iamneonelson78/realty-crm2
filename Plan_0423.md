# Plan — Route Cleanup, Feedback System, Email Branding, Listings Overhaul

## Context

Several pain points across the app need addressing in one coordinated change:

1. **Routes look bureaucratic.** Agent pages live at `/dashboard/listings`, `/dashboard/pipeline`, `/dashboard/connections`. URLs should be flat: `/dashboard`, `/listings`, `/pipeline`, `/connections`.
2. **Feedback capture is incomplete.** The `FeedbackWidget` only mounts inside `DashboardLayout`, so unauthenticated users on `/login` and `/signup` cannot submit. There are no attachments, no status tracking, and no admin page to triage feedback — it just writes to a table nobody can read.
3. **Approval email exists but branding is weak.** `api/notify-user-approved.js` already fires when admin flips a user to `active` ([AdminAccess.jsx:112](src/pages/admin/AdminAccess.jsx:112)), but emails use a bare text wordmark instead of the actual Realty CRM logo.
4. **Listings UX is awkward.** The primary button is "FB Post Generator" (the app's secondary job) instead of "Add Listings" (the primary job). There is no "Add Listing" form — users must open the generator to create a record. Row actions are cramped. Listings have no category, no filter/sort, no import/export, no print. Default FB templates are plain — not "FB worthy." Custom templates live in `localStorage` (per-browser, lost on device switch).

**Outcome:** clean URLs, feedback captured from anywhere with attachments and a real admin workflow, branded emails, and a listings page that delivers on the "killer feature" promise — fast to add properties, beautiful FB post templates with emoji/marketing flair, template persistence across devices, and spreadsheet-style list tooling.

## Decisions confirmed

- **File storage:** Google Drive via **service account auto-upload** (not Supabase Storage).
- **Drive layout:** two top-level folders — `Realty CRM / Feedback Attachments` and `Realty CRM / Listing Photos`. Subfolders auto-created per feedback ID / listing ID.
- **Feedback reply email:** yes — admin reply sends the user a branded email via a new `userFeedbackReplyEmail` template + `/api/notify-feedback-reply` route.
- **Listing photos:** included in this change, uploaded to Drive via the same service account.
- **Listing categories:** Condo, House, Apartment, Commercial, Lot, Other.

## Part 2 — Feedback everywhere + admin triage

### 2a. Schema (new migration file `supabase/migrations/2026XXXX_feedback_upgrade.sql`)

```sql
ALTER TABLE public.feedback
  ADD COLUMN status TEXT CHECK (status IN ('open','ongoing','replied','closed')) DEFAULT 'open',
  ADD COLUMN email TEXT,            -- captured when submitter is anonymous (login/signup)
  ADD COLUMN page_url TEXT,         -- where the feedback was submitted from
  ADD COLUMN user_agent TEXT,
  ADD COLUMN attachments JSONB DEFAULT '[]'::jsonb,  -- [{ path, name, size, mime }]
  ADD COLUMN admin_reply TEXT,
  ADD COLUMN replied_at TIMESTAMPTZ,
  ADD COLUMN updated_at TIMESTAMPTZ DEFAULT now();

-- Allow anonymous inserts (feedback on /login, /signup)
DROP POLICY IF EXISTS "Authenticated users can insert feedback" ON public.feedback;
CREATE POLICY "Anyone can insert feedback" ON public.feedback
  FOR INSERT TO anon, authenticated WITH CHECK (true);

-- Admins can read/update everything
CREATE POLICY "Admins read feedback" ON public.feedback
  FOR SELECT TO authenticated
  USING (EXISTS (SELECT 1 FROM public.profiles p WHERE p.id = auth.uid() AND p.role = 'admin'));

CREATE POLICY "Admins update feedback" ON public.feedback
  FOR UPDATE TO authenticated
  USING (EXISTS (SELECT 1 FROM public.profiles p WHERE p.id = auth.uid() AND p.role = 'admin'));
```

**Storage — Google Drive:** files go to `Realty CRM / Feedback Attachments / <feedback-id>/` via a shared service account (see Part 5). The `attachments` JSONB stores `[{ drive_file_id, name, mime, size, web_view_link }]`. Cap ~5 MB per file, 3 files per submission.

## Part 4 — Google Drive integration (shared infrastructure)

Shared by feedback attachments and listing photos.

### 4a. Google Cloud setup (one-time, documented in new `docs/GOOGLE_DRIVE_SETUP.md`)

1. Create a Google Cloud project (or reuse existing).
2. Enable **Google Drive API**.
3. Create a **service account**, download its JSON key.
4. In Google Drive UI, create the folder structure: `Realty CRM / Feedback Attachments/`, `Realty CRM / Listing Photos/`. Right-click each → **Share** → paste the service-account email (from the JSON) → give **Editor** access.
5. Copy each folder's ID from its Drive URL.
6. Add Vercel env vars:
   ```
   GOOGLE_SERVICE_ACCOUNT_KEY   = <full JSON as single-line string, Sensitive>
   GDRIVE_FEEDBACK_FOLDER_ID    = <id>
   GDRIVE_LISTINGS_FOLDER_ID    = <id>
   ```

### 4b. Server-side Drive wrapper

**New file:** `api/_lib/drive.js`

Uses `googleapis` npm package. Exposes:

```js
export async function uploadToDrive({ parentFolderId, subfolderName, file }) {
  // 1. Get-or-create subfolder under parentFolderId (cache by name)
  // 2. Upload file (multipart) with given name/mime, parents=[subfolderId]
  // 3. Set permission: role=reader, type=anyone (so the webViewLink works for admins/agents)
  // 4. Return { id, name, mimeType, size, webViewLink, webContentLink }
}

export async function deleteFromDrive(fileId) { ... }
```

Auth uses `google.auth.GoogleAuth` with `credentials: JSON.parse(process.env.GOOGLE_SERVICE_ACCOUNT_KEY)` and scope `https://www.googleapis.com/auth/drive.file` (only files created by this app, not the user's whole Drive — tighter security).

### 4c. Upload API routes (Vercel functions)

Two thin routes that call `uploadToDrive`:

- `api/feedback.js` — **POST** inserts feedback row + uploads attachments to `Realty CRM / Feedback Attachments / <feedback-id>/`. Accepts multipart/form-data. No auth header needed (anon submissions allowed), but rate-limit by IP with a simple in-memory counter or Upstash if available.
- `api/listing-photos.js` — **POST** uploads photos to `Realty CRM / Listing Photos / <listing-id>/`, returns URLs for the client to put into `listings.photo_urls`. Requires auth header; verifies caller is an agent.

Both respect the Vercel 4.5 MB body limit on Hobby / 100 MB on Pro — we cap at 5 MB per file client-side, max 3 files per request.

### 4d. Dependency

Add `"googleapis": "^144"` (or latest stable) to [package.json](package.json).

```sql
CREATE TABLE public.listing_templates (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  agent_id UUID REFERENCES public.profiles(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  body TEXT NOT NULL,
  created_at TIMESTAMPTZ DEFAULT now()
);
ALTER TABLE public.listing_templates ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Agents manage own templates" ON public.listing_templates
  FOR ALL TO authenticated
  USING (agent_id = auth.uid()) WITH CHECK (agent_id = auth.uid());
```
